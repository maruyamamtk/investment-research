## 1. システム概要

### 1.1 目的

日本株プライム市場（約1,600社）を対象に、以下の3段階の銘柄選定プロセスを自動化し、投資判断の補助情報を提供する。

| ステップ | 名称 | 内容 | 実行頻度 |
|---------|------|------|---------|
| ① | 監視対象銘柄の選定 | ファンダメンタルズスクリーニングにより全銘柄から監視対象リストを更新 | 週次 |
| ② | 購入銘柄の選定 | ①リストにテクニカル分析を実施し、BUYシグナルが発生した銘柄を購入候補リストに追加 | 日次 |
| ③ | 売却銘柄の選定 | ②リストにテクニカル分析 + ①と同等のファンダメンタルズ再確認を実施し、条件を満たさなくなった銘柄を除外 | 日次 |

**3ステップの関係**:
- 銘柄は「全市場 → ①監視リスト → ②購入候補リスト」の順に絞り込まれる
- ②購入候補リストはBUYシグナル発生時のみ追加され、チャート状況によっては長期間更新されない可能性がある
- ③では②リストの銘柄が条件を満たさなくなった場合に除外し、資産保全を図る

### 1.2 設計原則

- **再現性**: 同じ入力データに対して同じ結果を出力する定量ロジック
- **透明性**: シグナルの判定理由を自然言語で説明する（Claude AI）
- **保守性**: APIキーなしでも`--dry-run`モードで動作確認できる
- **コスト効率**: キャッシュにより不要なAPI呼び出しを抑制する

### 2.1 データソース

| ソース | 用途 | 取得方法 | 更新頻度 |

| Yahoo Finance `ticker.financials` | 損益計算書・貸借対照表・キャッシュフロー | yfinance ライブラリ | 週次キャッシュ（168h） |

| Yahoo Finance `ticker.calendar` | 次回決算発表予定日 | yfinance ライブラリ | キャッシュなし（毎回取得） |

### 2.2 対象市場

- 東京証券取引所 プライム市場（MarketCode: `0111`）
- 対象銘柄数: 約1,571社（J-Quants APIで取得）

- yfinanceはレート制限なしだが、非公式APIのため過度な並列リクエストを避ける
- バッチサイズ: 30銘柄/回、バッチ間インターバル: 2秒
- J-Quants `/fins/statements` はリフレッシュトークン → IDトークンの2段階認証が必要

## 3. 機能要件

### 3.1 ①監視対象銘柄の選定（週次スクリーニング）

毎週日曜日 8:00 に自動実行し、全プライム銘柄を2段階でスクリーニングして **監視対象銘柄リスト（`output/watch_list.md`）** を更新する。このリストが②・③の入力となる。

#### 統合総合スコアリング（全1,600社 → 上位20社）

**従来の個別ハードフィルタ（2-A〜2-F）およびStep1の除外フィルタを廃止し、全評価指標を1つの総合スコア（0〜100点）に統合する。** スコアが高い順に上位20社を選定する。

個別条件のpass/fail判定をなくすことで、「1つの軸は弱いが他の軸が突出している」銘柄も公平に評価できる。

**実装方式（2段階スコア計算）**

全1,600社に対してJ-Quants APIを逐一呼び出すと処理時間が過大になるため、取得コストの低い指標から先に計算し、候補を絞ってから詳細データを取得する。

| 段階 | 対象 | 使用データ | 目的 |
|------|------|---------|------|
| 段階1（速報スコア） | 全1,600社 | yfinance basic info のみ | 部分スコアで上位200〜400社に絞り込む |
| 段階2（精緻スコア） | 上位候補200〜400社 | J-Quants 財務諸表 + yfinance 詳細 | 全スコア次元を確定し最終順位を決定 |

**総合スコアリング表（全13次元・合計100%）**

| グループ | 評価次元 | 指標 | 重み | スコア基準（0〜10点） | 段階 | データソース |
|---------|---------|------|------|------|------|------|
| EPS成長 | 年次EPS成長率 | 直近3期の平均yoy成長率 | 13% | ≤0%→0点、≥30%→10点 | 2 | J-Quants |
| EPS成長 | 四半期EPS成長率 | 最新yoy成長率（単調増加で+1点ボーナス） | 12% | ≤0%→0点、≥25%→9点、+単調→10点 | 2 | J-Quants |
| 売上成長 | 年次売上高成長率 | 直近2期の平均yoy成長率 | 8% | ≤0%→0点、≥20%→10点 | 2 | J-Quants |
| 売上成長 | 四半期売上高成長率 | 最新yoy成長率 | 7% | ≤0%→0点、≥25%→10点 | 2 | J-Quants |
| 収益性 | ROE | 直近1期（当期純利益 / 自己資本） | 10% | ≤0%→0点、≥30%→10点 | 2 | J-Quants/yfinance |
| 収益性 | 営業利益率 | 直近期 | 7% | ≤0%→0点、≥25%→10点 | 1 | yfinance |
| CF健全性 | CF品質 | OCF / 純利益 | 8% | ≤0→0点、≥2.0→10点 | 2 | yfinance |
| CF健全性 | FCF継続年数 | 直近3年間のFCFプラス年数 | 5% | 0年→0点、3年→10点 | 2 | yfinance |
| 財務健全性 | 純負債/EBITDA | 直近期 | 10% | ≥5x→0点、≤0x→10点 | 2 | yfinance |
| 財務健全性 | 自己資本比率 | 直近期 | 5% | ≤10%→0点、≥50%→10点 | 1 | yfinance |
| バリュエーション | PEG比率 | PER ÷ 売上CAGR% | 10% | ≥5.0→0点、≤0.5→10点 | 1/2 | yfinance |
| 流動性 | 時価総額 | 直近時価総額 | 3% | ≤100億→0点、≥1兆→10点（対数スケール） | 1 | yfinance |
| 株主還元 | 配当性向 | 直近期 | 2% | ≥70%→0点、0%→10点 | 1 | yfinance |

> **欠損値の扱い**: データが取得できない次元は中間スコア（5点）を仮定し、スコアの安定性を確保する。

> **バリュエーション参考表示**: 総合スコア上位20社のEV/Revenue・EV/EBITDA・PER・PBRは適正レンジからの乖離をレポートに付記する（選定の除外条件ではない）。

**2-J. 定性分析チェック（Gemini AI）**

2-H バリュエーションチェック通過後の **Top 5銘柄**を対象に、Gemini AI が以下の5フレームワークに沿って定性評価を生成し、週次レポートに追記する。

数値スクリーニングが「現在の企業の健康診断」であるのに対し、定性分析は「将来の業績再現性・持続可能性のDNAチェック」として、中長期投資判断の補助情報を提供する。

#### 評価フレームワーク（5項目）

| # | 評価軸 | 主な確認ポイント |
|---|-------|----------------|
| Q1 | **事業モデルと競争優位性（Economic Moat）** | マネタイズ構造（ストック型 vs. フロー型）・参入障壁（特許・規制・スイッチングコスト）・ネットワーク効果・価格決定権の有無 |
| Q2 | **経営陣の質とガバナンス** | 創業者 or プロ経営者・過去のトラックレコード・中計ビジョンの一貫性・経営陣の自社株保有比率・IR開示の誠実さ |
| Q3 | **市場環境と成長ポテンシャル（TAM/SAM）** | 業界ライフサイクル（黎明期〜成熟期）・国策・規制の追い風・グローバル拡張可能性・DX/少子高齢化/脱炭素などメガトレンドとの整合 |
| Q4 | **顧客基盤とサプライチェーン** | 売上集中度（特定顧客依存リスク）・BtoBの解約率 / リピート率・調達先の地域・企業集中リスク |
| Q5 | **組織力と企業文化** | 優秀人材の獲得力・離職率・R&D投資比率・次の成長の種（新製品/新サービス）を仕込む文化の有無 |

#### 評価基準

各項目を 4段階 で評価する:

| ラベル | 基準 |
|--------|------|
| `Strong` | 明確な強みが存在し、競合優位が持続的 |
| `Moderate` | 一定の強みはあるが、課題も並存 |
| `Weak` | 優位性が希薄、または懸念材料が顕在 |
| `Unknown` | 公開情報が不足しており評価困難 |

各項目の評価に加え、根拠コメント（1〜2文）と **総合定性スコア（0〜10点）** を生成する。

#### 情報ソース（Gemini が参照する想定）

| ソース | 注目ポイント |
|--------|------------|
| 決算説明会資料・動画 | 社長自身の言葉の熱量、質疑応答での誠実さ・ロジックの明確さ |
| 中期経営計画 | 3〜5年後のビジョンと具体的な投資計画の整合性 |
| 有価証券報告書（【経営方針、経営環境及び対処すべき課題等】） | 企業自身が認識する最大リスク・ボトルネック |
| 業界レポート・専門誌 | 競合他社とのポジション比較、バリューチェーン構造 |
| 採用ページ・口コミサイト | 組織の風通し・開発力・営業力の実態 |

> **実装方針（v1.1）**: Gemini の一般知識（学習データ内の企業情報）を主な入力として使用する。v1.2 以降では EDINET / TDnet からの有報・決算資料の取得・要約を統合し、情報の鮮度と精度を向上させる。

#### 出力形式（`watch_list.md` への追記内容）

```
### 定性分析 — [銘柄名]

| 評価軸 | 評価 | 根拠 |
|--------|------|------|
| Q1 事業モデル・競争優位性 | Strong | サブスクリプション型の保守収益が全売上の60%を占め、スイッチングコストが高い。 |
| Q2 経営陣・ガバナンス | Moderate | 創業社長が保有比率30%を維持。中計は具体的だが進捗開示が年1回と頻度が低い。 |
| Q3 市場環境・成長ポテンシャル | Strong | DX需要と国内医療IT化の追い風を受ける成長市場。グローバル展開も模索中。 |
| Q4 顧客基盤・サプライチェーン | Moderate | 上位10社で売上の45%。サプライチェーンは国内主体で地政学リスクは低い。 |
| Q5 組織力・企業文化 | Weak | R&D比率は業界平均を下回り、エンジニア採用に課題を抱える兆候あり。 |

**総合定性スコア: 6.5 / 10**
```

#### Gemini AI 分析（総合スコア上位 Top5銘柄）

financial-services の `equity-research` パターンを参考にした分析内容を生成する。

- **投資テーゼ**: 強み・成長ドライバー・競争優位性（200〜300字）
- **ベアケース**: リスクスコア（0〜10）・主なリスク要因3点（地政学リスク・金利耐性・競合・財務脆弱性）
- **定性分析（2-J）**: 5フレームワーク（Q1〜Q5）の評価・根拠コメント・総合定性スコア（0〜10点）
- モデル: `gemini-2.5-flash`（コスト最適化）

### 3.2 ②購入銘柄の選定（日次テクニカル分析 → BUYシグナル管理）

平日 19:30（東証閉場後）に自動実行し、**①の監視対象銘柄リスト**を対象にテクニカル分析を行う。BUYシグナルが発生した銘柄を **購入候補リスト（`output/buy_candidates.md`）** に追加する。

> **注意**: チャートの状況によっては、BUYシグナルが発生せず購入候補リストへの追加が長期間行われない可能性がある。この場合、ウォッチ（中立）状態が継続する。

#### 使用するテクニカル指標

| 指標 | パラメータ | 計算ライブラリ |
|------|----------|-------------|
| 単純移動平均 (SMA) | 5日・20日・75日 | pandas（手実装） |
| 指数移動平均 (EMA) | 12日・26日 | pandas（手実装） |
| MACD | (12, 26, 9) | pandas（手実装） |
| RSI | 14日 | pandas（手実装） |
| ボリンジャーバンド | 20日・±2σ | pandas（手実装） |
| 出来高比率 | 当日 / 20日平均 | pandas（手実装） |

> `pandas-ta` がPython 3.9非対応のため、全指標を `src/technical/signals.py` に手実装している。

#### 市場全体の強弱判定（マーケットレジーム）

日次パイプライン実行時に市場指数（日経225: `^N225`）の日足データをダウンロードし、個別銘柄のBUY閾値を動的に変更する。

| 指標 | 内容 |
|------|------|
| 200日SMA比較 | 指数が200日SMA超: 中長期強気 / 割れ: 弱気 |
| 50日SMAの傾き | 20日間の変化率: 上昇トレンド / 下降トレンド |
| 50日SMA比較 | 指数が50日SMA超: 短期強気 / 割れ: 弱気 |

| レジーム | 判定条件 | BUY閾値 | 投資方針 |
|---------|---------|---------|---------|
| 🟢 強気（BULL） | スコア ≥ 3 | **3点** | 通常通りBUYシグナルを採用 |
| 🟡 中立（NEUTRAL） | -2 ≤ スコア ≤ 2 | **4点** | 強いシグナルのみBUY採用 |
| 🔴 弱気（BEAR） | スコア ≤ -3 | **5点** | ほとんどBUYを抑制（逆張り回避） |

> SELL閾値（3点）は市場レジームに関わらず固定。弱気相場ではSELLシグナルをより重視する。

#### チャートパターン認識

個別銘柄のOHLCVデータから以下のパターンを検出し、BUY/SELLスコアに加算する。

**強気パターン（BUYスコア加算）**

| パターン | スコア | 検出条件 |
|---------|-------|---------|
| ブレイクアウト | +3 | 200日高値を出来高1.5x以上で上抜け |
| ダブルボトム | +2 | 直近60日に同水準（5%以内）の底値2回、現在5%以上回復 |
| ブルフラッグ | +2 | 直近20日で10%急騰後、10日間のレンジが5%以内 |
| カップ・アンド・ハンドル | +2 | 90日のU字回復（深さ15〜35%）後のハンドル形成 |
| ブレイクアウト（出来高未確認） | +1 | 200日高値付近まで上昇（出来高確認なし） |

**弱気パターン（SELLスコア加算）**

| パターン | スコア | 検出条件 |
|---------|-------|---------|
| ヘッドアンドショルダーズ | +3 | 三山（中央最高・両肩10%以内）のネックライン割れ |
| ダブルトップ | +2 | 直近60日に同水準（5%以内）の高値2回、現在5%以上下落 |

#### シグナル判定ロジック

**🟢 BUY（強度スコア ≥ BUY閾値 AND BUY > SELL）**

以下の条件をスコアで加算し、3点以上かつBUY > SELLの場合:

| 条件 | スコア |
|------|-------|
| ゴールデンクロス発生（SMA5 > SMA20 に転換） | +3 |
| ゴールデンクロス維持 | +1 |
| RSI < 35 から反転上昇中 | +2 |
| RSI < 35（売られすぎ圏） | +1 |
| MACDヒストグラムがプラス転換 | +2 |
| MACDヒストグラムがプラス圏 | +1 |
| ボリンジャー下限ブレイク（反発期待） | +1 |
| 出来高比率 ≥ 1.2x（BUY優勢時に加算） | +1 |

**🔴 SELL（強度スコア3以上）**

| 条件 | スコア |
|------|-------|
| デッドクロス発生（SMA5 < SMA20 に転換） | +3 |
| デッドクロス維持 | +1 |
| RSI > 75 から反転下落中 | +2 |
| RSI > 75（過熱圏） | +1 |
| MACDヒストグラムがマイナス転換 | +2 |
| MACDヒストグラムがマイナス圏 | +1 |
| ボリンジャー上限ブレイク（過熱注意） | +1 |
| 出来高比率 ≥ 1.2x（SELL優勢時に加算） | +1 |

**🟡 HOLD（フェイクアウト回避）**

BUY/SELL条件を満たしていても、以下の場合はHOLDに上書きする:

- 次回決算発表日の **前後3日以内**（yfinance `ticker.calendar` で取得）
- BUYとSELLのスコアが拮抗している場合

**⚪ WATCH**

BUY/SELLスコアがともに3未満の場合。

#### フェイクアウト回避ルール（重要）

financial-services の「人間レビュー必須」原則に準拠し、以下のケースでは自動シグナルを抑制する:

1. 決算発表の前後3日間（誤ったシグナルが発生しやすい）
2. テクニカル指標のデータが30営業日未満の場合（指標の信頼性が低い）

#### ②の出力・更新ルール

| 条件 | アクション |
|------|---------|
| BUYシグナル（スコア≥3）かつフェイクアウト非該当 | `buy_candidates.md` に銘柄を追加（重複除外） |
| WATCH / HOLD | 監視対象リストでウォッチ継続。購入候補リストは更新しない |
| 同一銘柄が既に購入候補リストに存在する場合 | 重複追加せず、最新のシグナル日時・スコアを更新 |

### 3.3 ③売却銘柄の選定（テクニカル分析 + ファンダメンタルズ再確認）

平日 19:30（東証閉場後）に②と同じパイプライン実行内で処理する。**②の購入候補リスト** に掲載されている銘柄を対象に、売却すべきか否かを判定する。

#### 売却判定の2軸

**軸A: テクニカルSELLシグナル（日次）**

§3.2「シグナル判定ロジック」の🔴 SELLスコアが3以上かつ SELL > BUY の場合に、売却候補と判定する。

| 条件 | スコア |
|------|-------|
| デッドクロス発生（SMA5 < SMA20 に転換） | +3 |
| デッドクロス維持 | +1 |
| RSI > 75 から反転下落中 | +2 |
| RSI > 75（過熱圏） | +1 |
| MACDヒストグラムがマイナス転換 | +2 |
| MACDヒストグラムがマイナス圏 | +1 |
| ボリンジャー上限ブレイク（過熱注意） | +1 |
| 出来高比率 ≥ 1.2x（SELL優勢時に加算） | +1 |

**軸B: ファンダメンタルズ劣化チェック（週次・①と同等の条件）**

週次スクリーニング（毎週日曜）時に、購入候補リストの銘柄が①の監視対象条件を引き続き満たしているか確認する。以下のいずれかに該当した場合は売却候補と判定する。

| 確認条件 | 劣化の判断基準 |
|---------|-------------|
| Step1: 時価総額 | 100億円を下回った |
| Step1: 営業利益率 | 0%以下に転落 |
| Step1: 売上高成長率 | −10%を下回った |
| Step2-A: 年次EPS成長 | 直近期の成長率が25%を下回った or EPSがマイナス転落 |
| Step2-D: ROE | 15%を下回った |
| Step2-E: FCF品質 | OCF/純利益が0.8を下回った |
| Step2-F: 純負債/EBITDA | 3.0xを超過 |
| ①監視対象リストから除外された | 週次スクリーニングで上位20社から脱落 |

#### ③の出力・更新ルール

| 判定 | アクション |
|------|---------|
| テクニカルSELL（軸A）のみ該当 | 売却推奨としてLINE通知。`buy_candidates.md` から除外 |
| ファンダメンタルズ劣化（軸B）のみ該当 | 要注意フラグを付与。次回日次チェックでテクニカル確認。2回連続で条件劣化確認時に除外 |
| 両軸が同時に該当 | 即時除外 + LINE通知（優先度：高） |
| どちらも非該当 | 購入候補リストで保持継続 |

> **フェイクアウト回避**: テクニカルSELL時も、決算発表前後3日間はHOLDに上書きし除外しない（ただしファンダメンタルズ劣化による除外は決算前後でも実施する）。

## 4. 非機能要件

### 4.1 パフォーマンス

| 要件 | 目標値 |
|------|-------|
| 週次スクリーニング全体の実行時間 | 60〜120分（全1,600社対象時） |
| 日次シグナル検知の実行時間 | 5〜10分（20銘柄対象時） |
| APIキャッシュヒット時の再実行時間 | 1〜3分（週次）/ 1分（日次） |

### 4.2 信頼性・可用性

- パイプライン失敗時はログ（`logs/YYYYMMDD.log`）にエラーを記録し、正常終了した処理結果は保持する
- 個別銘柄のAPI取得失敗は `ERROR` シグナルとして出力し、他の銘柄の処理を継続する
- キャッシュ破損時は `--force-refresh` フラグで再取得できる

### 4.3 保守性

- APIキーなしでも `--dry-run` フラグで動作確認できる
- スクリーニング閾値はすべて `config/settings.yaml` で変更可能（コード修正不要）
- `--ticker` フラグで特定銘柄のみテスト実行できる

### 4.4 キャッシュの永続化（Cloud Run 対応）

Cloud Run Jobs のコンテナはエフェメラル（実行終了後に破棄）であるため、`cache/` ディレクトリへの書き込みは次回実行に引き継がれない。毎回フルスキャンが発生することで実行時間・コストが増大するため、以下の対応を行うこと（TODO 5-9）。

- `src/utils/cache.py` を拡張し、ローカルファイル読み書きに加えて **Cloud Storage バケット** への読み書きをサポートする
- 環境変数 `GCS_CACHE_BUCKET` が設定されている場合はGCS経由でキャッシュを読み書きし、未設定の場合はローカルファイルにフォールバックする（ローカル開発との後方互換性を維持）
- 期待効果: 週次ジョブの初回以降の実行時間を 60〜120分 → 1〜3分に短縮し、Cloud Run コストを大幅に削減する

## 5. システム構成要件

### 5.1 実行環境

| 項目 | 要件 |
|------|------|
| 実行基盤 | Google Cloud Run Jobs（サーバーレスコンテナ） |
| スケジューラ | Cloud Scheduler（タイムゾーン: Asia/Tokyo） |
| Python | 3.11以上（Dockerイメージ） |
| 必須パッケージ | yfinance ≥ 0.2.40、PyYAML ≥ 6.0、tabulate ≥ 0.9.0、google-genai ≥ 1.0.0 |
| ネットワーク | J-Quants API・Yahoo Finance・LINE API・Gemini API への HTTPS アクセス |
| GCP | プロジェクト作成済み・Cloud Run / Cloud Scheduler / Artifact Registry API 有効化済み |

> **選定理由**: macOS のローカル cron はPCが起動していないと実行されない。Cloud Run Jobs はコンテナをオンデマンドで起動・実行・停止するため、PC非依存で確実に定時実行できる。

### 5.2 スケジューリング（Cloud Scheduler → Cloud Run Jobs）

Cloud Scheduler のスケジュール式は **UTC** で記述する。JST（UTC+9）からの変換に注意すること。

| ジョブ | JST | UTC | cron式（UTC） |
|--------|-----|-----|--------------|
| 週次スクリーニング | 日曜 8:00 JST | 日曜 23:00 UTC（前日土曜） | `0 23 * * 6` |
| 日次シグナル検知 | 平日 19:30 JST | 平日 10:30 UTC | `30 10 * * 1-5` |
| LINE翌朝通知 | 平日 9:00 JST | 平日 0:00 UTC | `0 0 * * 2-6` |

> Cloud Scheduler の `--time-zone=Asia/Tokyo` オプションを使えばJSTで直接指定可能だが、UTC基準の方が環境依存を減らせる。

#### アーキテクチャ概要

```
Cloud Scheduler（cron式）
    ↓ HTTP POST（OIDC認証）
Cloud Run Jobs
    ├── weekly-job  → pipelines/weekly_pipeline.py
    ├── daily-job   → pipelines/daily_pipeline.py
    └── notify-job  → pipelines/notify_morning.py（将来実装）
```

#### 機密情報の管理

APIキー（Gemini・J-Quants・LINE）は `config/settings.yaml` にハードコードせず、**Secret Manager** に格納してCloud Runの環境変数として注入すること（**必須**）。

> **現状の問題（v1.1時点）**: `config/settings.yaml` に J-Quants・Gemini・LINE の認証情報が平文で保存されており、リポジトリへのアクセスができれば漏洩するリスクがある。TODO 5-8 にて対応予定。

移行手順の概要:
1. `gcloud secrets create` で各APIキーを Secret Manager に登録する
2. `deploy_cloud_run.sh` の `--set-env-vars` を `--set-secrets` に変更する
3. `config/settings.yaml` の認証情報キーを環境変数参照（`os.environ.get()`）に置き換える
4. `config/settings.yaml` および `config/settings.yaml.example` から実際の値を削除する
